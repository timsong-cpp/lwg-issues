4000. Eradicating explicit default constructors

Section: 26.6.4 [queue], 26.6.5 [priority.queue], 26.6.6 [stack], 29.6.3 [rand.eng], 29.6.6 [rand.device], 29.6.8 [rand.dist], 30.8 [string.streams], 31.10 [re.results], D.6.1 [depr.strstreambuf], D.16 [depr.conversions] Status: New Submitter: Tim Song Opened: 2017-10-09 Last modified: 2017-10-09 22:42:23 UTC

Priority: Not Prioritized

View all issues with New status.

Discussion:

Explicit default constructors block copy-list-initialization from {}. This typically arises when a function taking a T is given {} as the argument, when a T object is initialized with = {}, and – perhaps most invidiously – when an aggregate containing a T element is aggregate-initialized with no explicit initializer for that element (and no default member initializer): 

template<class X> struct aggregate { X x; };
aggregate<T> a{}; // ill-formed if T has an explicit default constructor.

For most Ts, this is quite surprising behavior. If T is a C++03 type, then this also presents a compatibility problem.

We should not use explicit default constructors without a good reason (such as for tag types - see LWG 2510), at least for classes that can conceivably be used in a copy-list-initialization context. LWG 2193 eliminated a number of such constructors for types with initializer-list constructors. The PR below finishes the job.

Proposed resolution:

This wording is relative to N4687.

[Drafting note: All facets have explicit default constructors, but they also have protected destructors and so can't be constructed in one of the contexts noted above. Even ignoring the access issue, a default-constructed facet is meant to be deleted by the locale, and new is always direct-initialization. Thus, no change is proposed for them. — end drafting note]
