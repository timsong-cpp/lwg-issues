3647. nothrow-input-iterator constraints should not mention copying

Section: 25.11.2 [special.mem.concepts] Status: New Submitter: Konstantin Varlamov Opened: 2021-11-30 Last modified: 2021-12-04 14:24:59 UTC

Priority: Not Prioritized

View all issues with New status.

Discussion:

25.11.2 [special.mem.concepts] states that a type qualifies as a nothrow-input-iterator only if no exceptions are thrown from, among other things, copy construction and copy assignment. However, being copyable isn't part of the requirements for an input_iterator on which nothrow-input-iterator is based (indeed, one of the things forward_iterator adds to input_iterator is copyability), and the nothrow-input-iterator concept doesn't add any new constraints related to copyability.

Proposed resolution:

This wording is relative to N4901.

  1. Modify 25.11.2 [special.mem.concepts] as indicated:

    template<class I>
concept nothrow-input-iterator = // exposition only
  input_iterator<I> &&
  is_lvalue_reference_v<iter_reference_t<I>> &&
  same_as<remove_cvref_t<iter_reference_t<I>>, iter_value_t<I>>;

    -2- A type I models nothrow-input-iterator only if no exceptions are thrown from increment, copy construction, move construction, copy assignment, move assignment, or indirection through valid iterators.
